# AGENTS.md

## 목적

이 저장소는 학습, 실험, 그리고 작은 재현 가능한 프로토타입을 위한 공간이다.  
추상화나 성급한 아키텍처 설계보다, **명확성**, **학습 가치**, **빠른 반복**을 우선한다.

## 저장소 작업 원칙

- 변경 작업 전 루트의 `README.md`와 대상 폴더의 문서를 먼저 읽는다.
- 한 번에 한 가지 주제만 다룬다.
- 개념을 명확하게 보여주는 **가장 작은 동작 예제**를 우선한다.
- 학습 과제와 무관한 리팩터링은 하지 않는다.
- 요구사항이 모호하면 큰 변경을 하기 전에 작업 계획을 먼저 요약한다.

## 저장소 구조

- 메인 솔루션 파일: `potato.study.sln`
- 학습 주제는 폴더 또는 프로젝트 단위로 분리한다.
- 권장 프로젝트 네이밍: `Potato.Study.<주제>`
- 예시:
  - `Potato.Study.Serializer`
  - `Potato.Study.Network`
  - `Potato.Study.Threading`

## 작업 스타일

- 영리한 코드보다 읽기 쉬운 코드를 우선한다.
- 재사용성이 충분히 증명되기 전에는 단순한 구조를 우선한다.
- 여러 접근 방식을 비교할 때는 먼저 최소 구현을 만들고, 이후 장단점을 문서화한다.
- 코드만으로 의도가 드러나지 않을 때만 주석을 추가한다.
- 학습 목표에 꼭 필요하지 않다면 새로운 의존성을 추가하지 않는다.
- 새 의존성을 추가했다면 왜 필요한지 설명한다.

## 학습 프로젝트 규칙

- 하나의 실험은 하나의 명확한 목표를 가져야 한다.
- 새로운 주제를 추가할 때는 짧은 `README.md`를 함께 작성한다.
- `README.md`에는 아래 내용을 포함한다.
  - 무엇을 실험하는지
  - 어떻게 실행하는지
  - 무엇을 배웠는지
- 일회성 코드나 탐색성 코드는 다른 주제와 섞지 말고 분리한다.

## 검증

- 변경한 프로젝트에 대해 가장 작고 적절한 빌드/테스트/확인 작업을 수행한다.
- 이미 문서화된 명령어가 있다면 그 명령어를 따른다.
- 명령어가 아직 없다면 솔루션/프로젝트 파일을 확인하고 가장 직접적인 로컬 명령어를 사용한다.
- 검증을 실행하지 못했다면 아래를 반드시 명시한다.
  - 무엇을 검증하지 못했는지
  - 왜 검증하지 못했는지
  - 다음에 무엇을 실행해야 하는지

## 작업 완료 보고 형식

작업을 마칠 때는 아래 내용을 보고한다.

- 무엇을 변경했는지
- 왜 변경했는지
- 어떻게 실행하거나 검증할 수 있는지
- 어떤 가정이나 제한사항이 있는지

버그 수정인 경우에는 아래도 함께 보고한다.

- 근본 원인
- 수정 방식
- 남아 있는 위험 요소

새 학습 예제를 추가한 경우에는 아래도 함께 보고한다.

- 이 예제가 어떤 개념을 보여주는지
- 이 접근 방식이 유용한 상황
- 이 접근 방식을 피해야 하는 상황

## 리뷰 기준

이 저장소의 변경 사항을 리뷰할 때는 아래를 우선적으로 본다.

- 설명하려는 개념이 올바르게 구현되었는지
- 나중에 다시 봐도 읽기 쉬운지
- 불필요한 추상화가 들어가 있지 않은지
- 숨겨진 부작용이 없는지
- 의존성이 과하게 늘어나지 않았는지
- 코드와 문서가 서로 어긋나지 않는지
- 실행/빌드 방법이 빠져 있지 않은지

샘플 실행 방법이 틀렸거나 문서가 오해를 부를 경우, 중요한 문제로 간주한다.

## 금지 사항

- 하나의 실험 때문에 저장소 전체를 갈아엎지 않는다.
- 명확히 오래된 자료가 아닌 이상, 노트/문서/스크래치 자료를 조용히 삭제하지 않는다.
- 구조 개선 효과가 분명하지 않다면 프로젝트 이동이나 이름 변경을 하지 않는다.
- 서로 관련 없는 개념을 하나의 샘플 프로젝트에 섞지 않는다.
- 목적 설명 없이 미완성 스캐폴딩만 남기지 않는다.

## 완료 기준

아래 조건을 만족하면 작업이 완료된 것으로 본다.

- 요청된 범위가 구현되었다.
- 관련 빌드/테스트/확인 작업을 수행했거나, 검증 공백이 명확히 기록되었다.
- 나중에 다시 보는 사람도 결과를 이해할 수 있다.
